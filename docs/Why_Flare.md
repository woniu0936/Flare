### 💡 为什么我们需要 Flare？(Why Flare?)

在现代 Android 架构（特别是 Jetpack Compose）中，处理**“只应发生一次的事件”**（One-off Events，如：登录成功后跳转、显示一次性 Toast）往往是开发者最容易踩坑的地方。

很多人在 ViewModel 中本能地选择 `Channel` 或 `SharedFlow` 来发送这类事件。表面上看，这很合理：ViewModel 存活时间长于 UI，事件流从 ViewModel 流向 UI。

**但这隐藏着一个致命的盲点：出问题的不是发送端（ViewModel），而是接收端（UI）！**

事件传递是一个“管道”。ViewModel 这头虽然一直活着，但 UI 那头在**屏幕旋转、配置更改、或退到后台**时，会断开连接、销毁，然后重新建立新的连接。就在 UI 断开连接的这短暂间隙，灾难就会发生。

让我们还原三个真实的“灾难现场”：

#### 💥 灾难 1：使用 SharedFlow (replay = 0) —— 幽灵事件，直接消失
1. 用户点击“登录”，触发耗时 2 秒的网络请求。
2. 用户随手将 App **切到后台**（此时 UI 停止监听）。
3. 第 2 秒，网络请求成功，ViewModel 极其敬业地执行 `sharedFlow.emit(Navigate)`。
4. SharedFlow 一看：“目前没有任何 UI 在监听我。” 于是，**这个跳转事件直接被丢弃（灰飞烟灭）**。
5. 用户切回前台，UI 重新连接。**结果：停留在登录页，什么也没发生，用户一脸懵逼。**

#### 💥 灾难 2：使用 SharedFlow (replay = 1) —— 无限重复回放的噩梦
为了解决事件丢失，你聪明地给 SharedFlow 加了缓存：`replay = 1`。
1. 网络成功，ViewModel 执行发射。因为在后台，SharedFlow 把事件**记在了缓存里**。
2. 用户切回前台，UI 重新连接，收到缓存的事件，**成功跳转到了首页**。（看起来很完美？）
3. **致命一击**：用户此时**旋转了屏幕**。
4. 屏幕旋转导致 UI 销毁并重建，新建的 UI **再次重新订阅**了 SharedFlow。
5. SharedFlow 热情地打招呼：“这是我缓存的最后一条消息：跳转到首页！”
6. **结果：UI 再次执行跳转！** 导致 Navigation 崩溃，或返回栈错乱。

#### 💥 灾难 3：使用 Channel —— 时机错乱与事件堆积
Channel 是一个队列，你不读，它就挂起存着。
1. 用户点击登录，觉得太慢，连按了 3 次，然后切到后台。
2. ViewModel 连续 3 次网络成功，执行了 3 次 `channel.send(Navigate)`。这 3 个事件全卡在管道里。
3. 用户切回前台，UI 重新连接。
4. **结果：管道瞬间开闸，UI 瞬间收到 3 个跳转事件，连续执行 3 次，App 当场崩溃！**

### 🛡️ Flare 的降维打击：状态驱动的终极答案

无论你怎么微调 `Channel` 或 `SharedFlow` 的参数，都无法完美处理 UI 断开连接时的那些“幽灵事件”。

Google 官方和现代架构指南给出的唯一正解是：**状态驱动（State-Driven）**。
*   **流（Flow/Channel）是动词**：试图在特定时间强制推一个动作，一旦 UI 错过，就会出 Bug。
*   **状态（State）是名词**：客观记录“现在可以跳转了”。UI 无论何时醒来，看到这个事实自己决定去动作；做完后，再亲自把这个事实抹掉（消费）。

**但这带来了另一个痛点：写起来太啰嗦了！** 你需要在 ViewModel 里增加无数的布尔值状态，并手动编写繁琐的“重置状态”逻辑。

#### 这就是 Flare 诞生的意义！

**Flare** 完美融合了 **“Channel 的极简易用性”** 和 **“状态驱动的绝对安全性”**。

它借鉴了流式处理的 **Cursor（游标）机制**：
1. **发射（Fire）**：ViewModel 依然像用 Channel 一样，无脑发射事件（Flare 内部会自动为其打上递增的唯一 ID）。
2. **缓存（Replay）**：Flare 底层使用带缓存的 SharedFlow，绝不丢失后台事件。
3. **消费与游标（Cursor）**：ViewModel 内部替你维护了每个消费者的游标。当 UI 醒来（或屏幕旋转重建）收到回放事件时，Flare 会自动拦截：“这个 ID 的事件你已经处理过了，忽略它！”

**有了 Flare，你只需专注业务，彻底告别生命周期错位带来的恐惧！**
